# Histometer Design Document

> **Specification note:** The current operational domain model and workflow decisions are defined in [phase_zero_spec.md](phase_zero_spec.md). Where the documents conflict, the phase-zero specification is authoritative. This older document remains useful for product history and the original MVP concept.

## 1. Purpose

Histometer is a Windows desktop application for tracking histology samples from intake through processing, sectioning, staining/IHC, imaging, and analysis. The application should replace the current manually maintained spreadsheet workflow while preserving an underlying spreadsheet-shaped record for reporting, backup, and possible SharePoint synchronization.

The current prototype is a local Tkinter + SQLite desktop app. The first usable version should make the workflow more rigorous than the existing CSVs by separating:

- Project identity
- Individual samples
- Pre-processing events
- Processing and sectioning status
- Stain/IHC requests
- Imaging status
- Analysis status

## 2. Current Spreadsheet Workflow

The current files represent three related work queues:

- Master log: one row per sample/specimen, with a sequential legacy sample number and flexible notes.
- Processing progress: batch-level tracking for fixation, decalcification, processing, embedding, and sectioning.
- Analysis progress: batch-level tracking for stains/IHC, due dates, pictures, start dates, and completed dates.

The app should retain the useful structure of these sheets, but reduce ambiguity by making each workflow event explicit and timestamped.

## 3. Sample Numbering

The current sample numbering system is a legacy sequential letter plus number sequence. It started around `A-0001` and has reached `S` in the mid-2500s. This is not ideal because the identifier does not encode project context.

The new system should generate sample IDs from the project abbreviation:

```text
EE-0001
EE-0002
EE-0003
```

Where:

- `EE` is the project abbreviation, such as Enthesis Engineering.
- The number is sequential within that project.
- Sample IDs are generated by the app, not manually typed by the user.

Projects therefore need a required project abbreviation/code.

## 4. Core Workflow

### 4.1 Project Setup

Each project should store:

- Project abbreviation/code
- Project name
- Team lead
- Status: active or inactive

Only active projects appear in the left sidebar by default.

### 4.2 Sample Intake

When a sample is added to a project, the app should automatically generate the next sample ID and record the intake date. The user should be prompted for:

- Description
- Fixative: `PFA`, `Z-Fix`, or other
- Processing run: short or long
- Whether decalcification is needed before fixation
- Sectioning/cut notes
- Slide notes
- Flexible notes

The legacy `Date (Fixed)` field should be interpreted as the date the sample was placed in fixative.

### 4.3 Pre-processing Stage

Before short or long processing, samples must move through a pre-processing checklist:

1. Decalcification complete, if the sample requires decalcification
2. Placed in fixative
3. Removed from fixative
4. Placed in ethanol
5. Begun processing

When a user checks one of these steps, the app should automatically record the timestamp. Samples appear in a pre-processing list until processing begins.

The app should not allow a sample marked as requiring decalcification to move into fixation or later stages until `Decalcification Complete` has been timestamped.

Current assumption: samples go from ethanol directly into processing.

### 4.4 Processing and Sectioning

After pre-processing, samples move through:

1. Processing
2. Processor pickup
3. Needs embedding
4. Embedded inventory
5. Needs sectioning
6. Sectioned

Processing runs are timed from the `Begun processing` timestamp. Short runs are 18 hours and long runs are 52 hours. Once the run time has elapsed, samples should move automatically to `Processor Pickup`; users can also manually drag a sample to pickup if the physical run is complete before the next refresh.

`Sectioned = Yes` means slides physically exist and are ready for staining. Embedded samples should not automatically move to sectioning. They should first live in an embedded inventory and only move to `Needs Sectioning` when a user decides to cut them.

### 4.5 Stains, IHC, Imaging, and Analysis

Stain/IHC tracking should be separate from general completion status because the current `Completed Date` field is ambiguous.

The downstream workflow should be:

1. Stain/IHC requested
2. Stain/IHC started
3. Stain/IHC completed
4. Pictures needed
5. Pictures taken
6. Analysis complete

Pictures should be treated as a distinct workflow stage after staining/IHC.

## 5. User Interface Concept

### 5.1 Main Window

The primary screen should be a workflow board, not a spreadsheet-style table. It should show all active, not-yet-analyzed samples in two left-to-right workflow lanes.

Processing and embedding lane:

1. Pre-processing
2. Processing
3. Processor Pickup
4. Needs Embedding

Embedded inventory and analysis lane:

1. Embedded Inventory
2. Needs Sectioning
3. Staining / IHC
4. Pictures / Analysis Pending

The desktop layout should have these working areas:

- Left sidebar: simplified active project picker with a compact add button and project manager access
- Main board: all unfinished samples across active projects
- New sample dialog: add a new sample to pre-processing using the selected project's generated ID prefix
- Details panel: hidden by default and opened as a right-side drawer so it does not cover the workflow board
- Toolbar actions: create sample, undo, save, show details, mark analyzed, and refresh
- Drag-and-drop cards: users move samples between queues directly; movement buttons are intentionally removed
- Drag preview: the tile being dragged follows the cursor as a translucent mini-card
- Card layout: main workflow queues can show two cards per row

### 5.2 Work Queues

The app should eventually provide queue views for:

- Pre-processing
- Processing
- Processor pickup after processing
- Needs embedding
- Embedded inventory
- Needs sectioning
- Stain/IHC in progress
- Pictures needed
- Analysis pending
- Complete

Queue views should be driven by timestamps and workflow fields, not by manual spreadsheet sorting.

Processing run timing should be visible in the board:

- Short runs: 18 hours
- Long runs: 52 hours

When processing is completed, the sample should move to `Processor Pickup` so staff know it requires physical retrieval from the processing lab.
Users should also be able to manually move a sample from `Processing` to `Processor Pickup` if the physical run has completed before the app refreshes.

Embedded samples should not automatically proceed to sectioning. They should land in an embedded inventory first, then be moved into `Needs Sectioning` only when the user decides those embedded samples should be cut.
The embedded inventory should be compact: sample ID, description, and cut notes, without large slide cards. It should include a project filter dropdown with `All Projects` as the default.

Cards should avoid spreadsheet-like labels where possible. For example, a cut note should display as the note itself rather than `Cut notes | note text`.

## 6. Data Model

### 6.1 Projects

| Field | Type | Notes |
| --- | --- | --- |
| id | Integer | Primary key |
| code | Text | Project abbreviation, such as `EE` |
| name | Text | Human-readable project name |
| team_lead | Text | Lead or owner |
| is_active | Boolean | Controls sidebar visibility |
| created_at | Timestamp | Audit field |

### 6.2 Samples

| Field | Type | Notes |
| --- | --- | --- |
| id | Integer | Primary key |
| project_id | Integer | Foreign key to projects |
| project_sample_number | Integer | Sequential within project |
| sample_code | Text | Generated code, such as `EE-0001` |
| sample_description | Text | Human-readable sample description |
| date_added | Date | App entry date |
| fixative_agent | Text | `PFA`, `Z-Fix`, or other |
| needs_decalcification | Boolean | Whether decalcification is required |
| processing_type | Text | Short or Long |
| cut_notes | Text | Sectioning or orientation instructions |
| slide_notes | Text | Slide-specific notes |
| stains | Text | Requested or completed stains/IHC |
| overall_notes | Text | Flexible notes |
| current_stage | Text | Latest workflow stage |
| decalc_completed_at | Timestamp | Decalcification completed before fixation |
| *_at | Timestamp | Other timestamp columns for workflow events |
| created_at | Timestamp | Audit field |

### 6.3 Future Tables

The prototype currently keeps stain requests in a text field. A fuller implementation should split these into separate tables:

- `processing_batches`
- `stain_requests`
- `imaging_requests`
- `workflow_events`

This will support sample batches, multiple stains per sample, and a proper audit trail.

## 7. Spreadsheet and SharePoint Strategy

The local database should be the operational source of truth for the executable. It can then generate or sync spreadsheet-compatible tables.

Recommended phases:

1. Local SQLite app with CSV/XLSX export.
2. Import legacy CSVs into the new project/sample model.
3. Export a normalized Excel workbook with sheets for projects, samples, processing, stains/IHC, imaging, and analysis.
4. Add SharePoint synchronization after the lab decides whether to use a local OneDrive-synced workbook path or Microsoft Graph authentication.

SharePoint Excel sync should be treated as an integration layer because it requires permissions, authentication, and conflict handling.

## 8. MVP Scope

The current prototype includes or is implementing:

1. Add and manage projects with project abbreviations.
2. Generate project-scoped sample IDs automatically.
3. Capture fixative, processing type, decalcification requirement, description, cut notes, slide notes, and flexible notes at intake.
4. Show a two-lane workflow board for all open samples.
5. Timestamp checklist events for decalcification completion, fixation, ethanol, and processing start.
6. Automatically move completed short/long processing runs to `Processor Pickup` after 18 or 52 hours.
7. Allow drag-and-drop movement between workflow queues with a cursor-following drag preview.
8. Keep details hidden by default in a right-side drawer.
9. Keep embedded samples in a compact inventory before they are chosen for sectioning.
10. Preserve flexible notes.

Remaining MVP work:

1. Export or sync the underlying tabular record.
2. Import legacy CSVs into the new project/sample model.
3. Package the app as a Windows executable.

## 9. Open Questions

- Should users be able to correct a mistaken timestamp, and should corrections require initials or a reason?
- Do short and long processing ever both apply to one sample, or should that become two linked processing requests?
- Should decalcification capture only completion, or should it have separate start and end timestamps?
- Should stain/IHC requests be entered at sample intake, after sectioning, or both?
- Should imaging mean microscope pictures, slide scans, or either?
- Which SharePoint account or lab identity should own the synced workbook?
